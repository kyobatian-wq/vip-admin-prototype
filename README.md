# DD.GG VIP系统迭代需求文档
### （对标Panda后台VIP管理功能）

---

## 一、需求背景

当前 DD.GG 后台的 VIP 系统功能较为基础，仅支持简单的等级配置和日/周/月礼金发放，缺少多商户管理、精细化流水稽核、保级降级、未领取奖励处理规则、以及页面视觉自定义等能力。为提升VIP体系的运营灵活性和风控精度，现参照 Panda 后台的 VIP 管理模块，对 DD.GG 的 VIP 系统进行迭代升级。

---

## 二、迭代目标

将 DD.GG 现有的单一、内联编辑式 VIP 配置，升级为**多商户可管理、条件精细化、带风控稽核、支持保级降级机制、且前端展示可自定义**的完整VIP体系。

---

## 三、详细需求清单

### 3.1 多商户/多地区管理能力

**现状**：DD.GG 无商户/地区维度，所有配置全局生效。

**需求**：
- 页面右上角增加「地区」「商户」下拉选择器
- 支持按地区+商户维度独立配置VIP等级体系、礼金规则、稽核规则
- 不同商户之间的VIP配置数据相互隔离，互不影响

---

### 3.2 VIP总开关与展示控制

**需求**：
- 新增「VIP系统总活动开关」
- **本期范围**：总开关固定为「开启」状态并锁定，前端置灰/加锁形图标展示，**不支持运营手动关闭**；点击开关无任何响应（不弹窗、不改变状态），鼠标悬停提示"本期固定为开启状态，暂不支持关闭"
- 「清除数据/保留数据」二次确认弹窗为**后续版本**支持关闭功能时的预留设计，本期不实现该交互
- 新增「是否在活动列表中展示」选项（是/否），控制该VIP活动是否对前台用户可见

---

### 3.3 编辑模式调整

**现状**：DD.GG 等级表格为内联编辑（直接在表格输入框中改数值），风险较高，容易误操作。

**需求**：
- VIP等级表格默认为**只读展示**
- 增加独立「编辑」按钮，点击后进入编辑弹窗/编辑态进行修改，保存后统一提交
- 降低误触、误改的风险，操作留痕更清晰

---

### 3.4 升级条件颗粒度细化

**现状**：仅支持"累计存款""累计有效流水"两个开关作为统一升级条件。

**需求**：
- 每个VIP等级单独设置「晋级需充值」「晋级需流水」两个字段（数值可分别配置，不再是全局开关）
- 支持「晋级奖励」独立配置

---

### 3.5 日/周/月礼金的独立门槛设置

**现状**：日/周/月礼金仅通过统一开关勾选控制发放与否，无各自独立的流水门槛。

**需求**：
- 日奖励、周奖励、月奖励**分别增加"所需流水"字段**，即每类奖励可单独设置需要达到的流水量才能领取
- 三类奖励的开放状态仍保留独立勾选开关（日奖励/周奖励/月奖励），勾选后奖励生效；不勾选则不发放且前台不展示（该提示文案沿用）

---

### 3.6 保级（降级）机制 【新增功能】

**现状**：DD.GG 无保级/降级概念，用户升级后等级永久保留。

**需求**：
- 新增「保级开关」（开启/关闭）
- 开启后，每个VIP等级需配置「保级需充值」「保级需流水」
- **判定时点与周期**：每月1号进行保级判断，统计**上月（自然月）**的累计充值与累计有效流水
- 规则：上月充值和流水同时达到保级所需条件时，视为保级成功，维持当前等级；否则**降1级**（不满足保级条件时，等级只下降一级，不会跌至0级或按比例跌落）

---

### 3.7 未领取奖励处理规则 【新增功能】

**需求**：新增「领取规则」配置项，支持以下几种模式（单选）：

1. **跨天/跨周/跨月，未领取奖励邮件下发**
2. **跨天/跨周/跨月，未领取奖励系统回收作废**
3. **受众用户1**：跨天/跨周/跨月，未领取奖励系统回收作废，且需配置"N天内充值"作为受众用户判定条件，**N天通过下拉框选择**，可选项为：5天内充值 / 7天内充值 / 10天内充值 / 15天内充值 / 20天内充值 / 30天内充值 / 40天内充值（不支持自定义输入天数，仅在以上档位中选择）
4. **受众用户2**：跨天/跨周/跨月，不满足受众条件不派发奖励，未领取奖励系统回收作废

> 说明：该模块用于控制过期未领取的VIP奖励如何处理，避免奖励无限堆积或流失用户权益不清晰的问题。

---

### 3.8 流水稽核与限定游戏平台 【新增风控功能】

**现状**：DD.GG 的流水计算无稽核倍数、无游戏平台限定，存在被用户利用低倍率游戏"刷"等级/流水的风险。

**需求**：
- 新增「稽核倍数」字段，**仅支持设置整数**。稽核倍数用于确定各类奖励（升级礼金/日礼金/周礼金/月礼金）对应需要考核的流水量，计算逻辑为：**所需稽核流水 = 奖励金额 × 稽核倍数**。例如稽核倍数设置为1倍：奖励50，则需考核流水50；设置为2倍：奖励50，则需考核流水100（用户需完成对应倍数的流水才能领取该笔奖励）。**稽核倍数为全局统一配置**，同一商户/地区下所有VIP等级共用同一个倍数，不支持按等级单独设置
- 新增「稽核限定平台」选项：
  - **不限制**：全平台流水均计入考核
  - **仅限以下平台**：可勾选具体的游戏分类（电子/体育等）及分类下的具体游戏平台（如 FaChai、PLAYNGO、TOPPLAYER、PANDA、FASTSPIN、inout 等），仅勾选的平台流水计入VIP升级/保级/礼金考核
- 支持"全部平台"一键全选/取消

---

### 3.9 规则说明文案配置

**需求**：
- 新增「规则说明」模块，分两个子标签：
  - **日周月/晋级奖励**说明
  - **保级说明**
- 每类说明支持「规则类型」选择：**默认**（展示系统预置文案，只读）或**自定义**（切换后显示**富文本编辑器**，支持加粗/斜体/下划线/有序或无序列表/插入超链接/清除格式，运营可自行编辑文案内容与排版）

---

### 3.10 VIP页面视觉自定义 【新增功能】

**需求**：
- 新增「背景样式」选择：**系统默认** 或 **自定义**
  - 选择自定义时，支持上传背景图（支持JPG/PNG/SVG格式，建议尺寸 660×220）
- 上传后提供「宣传图预览」区域，实时展示当前配置效果，便于运营在上线前确认视觉效果

---

## 四、功能对比总览表

| 功能模块 | DD.GG 现状 | 迭代后目标 |
|---|---|---|
| 商户/地区管理 | 无 | 支持按地区+商户独立配置 |
| VIP总开关 | 无 | 本期固定为「开启」并锁定（不支持关闭）+ 活动列表展示控制 |
| 等级表编辑方式 | 内联直接编辑 | 只读展示 + 独立编辑弹窗 |
| 升级条件 | 存款/流水二选开关 | 晋级需充值、晋级需流水独立字段 |
| 日/周/月礼金门槛 | 无独立门槛，仅开关控制 | 各自独立"所需流水"字段 |
| 保级/降级机制 | 无 | 新增保级开关+充值/流水双条件判定 |
| 未领取奖励处理 | 无 | 4种领取规则可选，支持受众用户细分 |
| 流水稽核 | 无 | 稽核倍数折算 + 限定游戏平台计入 |
| 规则说明文案 | 无 | 默认/自定义文案，日周月与保级分开配置 |
| 前端视觉自定义 | 无 | 富文本编辑（自定义文案）、背景图上传、宣传图预览 |

## 五、需求确认结论

以下事项已与产品/运营对齐，结论如下：

| 序号 | 事项 | 结论 |
|---|---|---|
| 1 | 保级失败后的降级逻辑 | **降1级**（不满足保级条件时只降一级，不跌至0级或按比例跌落） |
| 2 | 稽核倍数配置粒度及计算逻辑 | **全局统一配置**，仅支持整数；计算逻辑为「所需稽核流水 = 奖励金额 × 稽核倍数」（如1倍：奖励50需稽核流水50；2倍：奖励50需稽核流水100），不支持分等级单独设置 |
| 3 | 领取规则"N天内充值"的可选范围 | **下拉框选择**，档位为：5/7/10/15/20/30/40天内充值，不支持自定义天数 |
| 4 | 存量VIP用户数据是否需要迁移/重新核算 | **不需要**，迭代上线后按新规则从当前数据状态开始计算 |
| 5 | 保级判断时点及统计周期 | 每月1号进行保级判断，统计**上月（自然月）**充值、流水 |
| 6 | 总开关本期是否支持关闭 | **不支持**，本期固定为「开启」并锁定；「清除数据/保留数据」二次确认弹窗为后续版本预留设计 |

> 补充说明：项目当前尚未上线，无存量用户数据，因此**不需要**设计降级过渡期。

---

## 六、优先级建议

| 优先级 | 功能模块 |
|---|---|
| P0（核心，建议优先） | 编辑模式调整、日/周/月独立流水门槛、保级机制 |
| P1（重要） | 流水稽核与限定平台、未领取奖励处理规则 |
| P2（体验优化） | 多商户地区管理、规则说明文案自定义、前端视觉自定义 |

> 注：多商户/地区管理若DD.GG本身已有全局商户体系支撑，可能涉及底层架构改动，建议单独评估工作量，不排除需要拆分为独立项目。
