1. 背景与目标

商户在提现（出款）业务中涉及多个环节：会员发起提现 → 系统/人工审核 → 代付渠道出款 → 出款结果回写。 为提升处理效率、降低人工审核成本、控制资损风险，本次需求围绕"出款审核 + 自动代付"链路， 对以下 5 个后台页面进行统一设计：

  • 待审核出款订单：审核员处理会员提现申请的主战场，支持锁定、拒绝、罚没、人工完成、代付等操作；
  • 已审核出款订单：已完结订单的查询与追溯；
  • 总出款记录：全量出款流水查询，供财务对账使用；
  • 提现管理（自动代付配置）：配置自动代付的开关、金额范围、失败驳回策略、渠道上下架规则；
  • 代付商户管理：维护接入的第三方代付渠道基础信息。
目标：在保证资金安全（锁定机制、余额校验、备注留痕）的前提下，让符合条件的小额订单可以无需人工干预自动完成代付， 人工审核员只需处理异常订单，从而降低整体处理时长与人力成本。

2. 名词与范围

名词说明

4.4 代付流程

  1. 点击「代付」（支持单笔或勾选多笔「批量代付」），弹出代付弹窗，展示所有选中订单的订单信息表格；
  2. 下方展示可选代付渠道列表，每个渠道显示当前余额；余额为 0 的渠道置灰不可选，可点击「刷新」重新拉取余额；
  3. 系统实时计算所选订单的实付总额，与当前选中渠道余额比较：若余额不足，「出款」按钮置灰禁用，并在按钮上方提示"代付渠道余额不足（需要 X，渠道名 余额 Y），请更换代付渠道"；
  4. 余额充足时点击「出款」，生成代付单号，订单状态置为「代付中」，弹出代付结果表（支持多笔订单逐行展示成功状态）；
  5. 后续通过「更新订单状态」拉取渠道回执，订单状态最终收敛为「代付成功 / 代付失败 / 代付异常」。
批量代付时，校验条件为「勾选订单全部已锁定 且 状态均为 待审核/审核中」，否则拦截并提示"存在未锁定或状态不可代付的订单，请检查勾选项"。

4.5 批量操作

工具栏提供：批量锁定 / 批量解锁 / 批量拒绝 / 批量罚没 / 批量代付 / 批量异常处理。其中批量拒绝、批量罚没复用单笔的备注弹窗（作用于所有勾选订单），批量代付复用 §4.4 流程。

4.6 审核类型判定逻辑（免审不通过 vs 自动代付条件不符）

「审核类型」字段用于标注一笔订单为何最终落在「待审核出款订单」列表、需要人工介入。系统对每笔提现订单依次经过两道判断关卡， 两道关卡各自失败会产生不同的审核类型，处理方式也不同：

  1. 第一关：是否命中免审风控条件——由「提现管理 → 提现条件配置」页的「免审风控开关」控制（详见 §7.1「免审风控开关」配置项）：
    • 免审风控开关关闭：第一关不生效，所有订单直接视为"满足免审条件"，跳过第一关，直接进入第二关判断；
    • 免审风控开关开启：系统判断会员是否命中「已选会员层级」中的任一层级标签——命中则判定为 免审不通过，订单转人工审核；未命中则视为满足免审条件，进入第二关。
  2. 第二关：是否满足自动代付条件（「自动代付开关」是否开启、金额是否落在「自动代付金额范围」闭区间内、是否命中「首次出款强制人工审核」）
    → 满足：系统自动发起代付，无需人工介入；不满足：判定为 自动代付条件不符，订单进入人工队列等待手动代付
关于"免审风控开关关闭"的简化结论：如果商户没有开启「免审风控开关」（即不需要按会员层级做人工审核拦截）， 那么第一关对所有订单都直接放行，系统只需要判断第二关——自动代付条件即可决定订单是自动代付还是转人工（自动代付条件不符）。 换句话说，「免审风控开关」关闭时，「免审不通过」这个审核类型不会再出现，待审核列表里因风控进入的订单会归零， 列表中剩下的人工订单全部是「自动代付条件不符」。
审核类型触发的关卡常见触发场景审核员应做什么
免审不通过 第一关（免审风控） 「免审风控开关」处于开启状态，且该会员被打上了「已选会员层级」中的某个标签 （第一期免审风控条件仅支持按会员层级判断，后续可扩展账号风险分、绑定手机/实名等其他维度） 需要认真核实订单本身是否存在风险（会员身份、账户信息、行为异常等），核实无误后再决定通过/代付，或拒绝/罚没
自动代付条件不符 第二关（是否可自动出款） 提现金额超出「自动代付金额范围」/ 自动代付总开关被关闭 / 命中「首次出款强制人工审核」 该笔订单本身已通过第一关（或第一关未生效），审核员一般只需核对信息无误后手动点击「代付」即可，无需重新做风险判断
关键区别：「免审不通过」卡在"要不要审核"这一关，需要审核员认真判断订单风险；「自动代付条件不符」是第一关已经放行（无论是因为未命中风控层级，还是因为免审风控开关本身就关闭）， 只是卡在"能不能自动打钱"这一关，审核员通常只需要手动完成代付操作。两者都会让订单出现在「待审核出款订单」列表中， 但对应的「操作备注」应分别记录是"审核未通过的具体原因"还是"未自动代付的具体原因"，便于后续数据统计与问题排查。

7.1 配置项清单

配置项控件业务规则
免审风控开关 （新增，位于自动代付开关上方） 开关（默认开） + 会员层级多选 用于具体定义模块一 §4.6 提到的「免审条件」——第一期免审风控条件仅支持按会员层级判断：
· 开启：可通过「修改」按钮弹出的多选弹窗，勾选一个或多个会员层级；会员一旦被打上所选任一层级标签，其提现订单直接判定为「免审不通过」，强制转人工审核；
· 关闭：该判断关卡不生效，所有订单都视为满足免审条件，直接进入「自动代付条件」判断（即系统此时只需要判断自动代付相关规则）。
层级多选采用「只读展示 + 修改按钮 + 弹窗勾选」的交互，避免误触；未选择任何层级时展示为灰色「未选择任何层级」提示。
自动代付开关开关（默认开） 开启：符合免审条件 且 金额落在自动代付范围内的订单，系统自动发起代付，无需人工介入；

自动代付金额范围最小金额 ~ 最大金额 仅在此区间内的订单才会触发自动代付；超出范围的订单转人工审核（对应模块一中「审核类型：自动代付条件不符」）。
边界值口径（已确认）：区间为闭区间，提款金额等于最小值或最大值时均计入自动代付范围（即 [最小金额, 最大金额]）。
首次出款强制人工审核开关（新增，默认开） 开启后，会员的首次出款订单（列表「是否首次出款」= 是）一律转人工审核，不受自动代付金额范围限制，即便金额在自动代付区间内也不会自动代付； 关闭后，首次出款订单与其他订单一样按金额范围规则判断。
（§12 已确认：需要该开关，UI 位置见本页「提现条件配置」区块）
代付失败自动驳回开关（默认开） 代付订单轮询所有满足出款金额条件的代付渠道，若全部尝试均失败，订单自动拒绝，提现金额退回会员账户，无需人工介入。
失败判定口径（已确认）：本期不做轮询重试——只要本轮所有可用渠道尝试均返回失败，立即判定为失败并自动驳回，不设置额外的轮询次数上限或超时等待。
自动代付渠道配置渠道列表（开关 + 下架规则下拉） 逐个渠道配置：① 是否上架用于自动代付；② 自动下架标准：不自动下架（推荐）/ 余额低于阈值时自动下架（需填余额阈值）/ 连续失败超过次数时自动下架（需填次数阈值）。
渠道被禁用/下架时的联动规则（已确认）：若该渠道下仍存在「进行中」（代付中）的订单，这些订单状态统一置为「代付异常」，转人工核实是否需要重新补单，不做自动切换到其他渠道重试。
产品强调：建议将下架标准默认配置为"不自动下架"。如果渠道被自动下架、又恰好是唯一可用渠道，会导致自动代付无渠道可用， 大量订单被迫转入人工队列，产生积压。研发需要在配置页面用醒目文案（红字）提示该风险，本 PRD 也已在原型中体现。

7.2 保存/重置

页面底部提供「保存」「重置」按钮。保存时校验：若「自动代付开关」为开，则金额范围两个字段均为必填且最小值需小于最大值（原型暂未做该项拦截，建议开发阶段补充）。

9.2 消息提示（Toast）

所有非阻断性操作反馈（查询完成、复制成功、批量操作完成、状态刷新等）统一使用顶部居中的黑底浮层提示，2 秒左右自动消失，不打断用户当前操作。

9.3 备注双轨制

凡涉及资金处置（拒绝 / 罚没 / 人工完成）的操作，均要求同时填写两类备注：

备注类型可见范围
操作备注会以邮件形式发送给会员，并展示在会员端「提款详情」页面
订单备注仅在后台内部可见，不对会员展示

9.4 锁定机制

锁定的目的是防止多名审核员同时处理同一笔订单造成的重复操作或资金重复处理风险。锁定后的订单，只有「锁定操作人」与「拥有最高权限的账号」可以继续操作；其余账号即便有权限查看列表，也无法对该订单进行拒绝/罚没/代付等处置操作，只能查看明细。

10. 数据字典 / 状态枚举

10.1 订单状态（待审核出款订单）

状态含义可执行操作
待审核已提交，尚未进入人工或系统审核锁定 → 处理/拒绝/罚没/代付
审核中已进入审核流程但未决策（含自动代付条件不符被打回的订单）同上
代付中已发起代付请求，等待三方渠道回执更新订单状态
代付成功三方渠道确认打款成功更新订单状态（终态，随后流转至已审核出款订单）
代付失败三方渠道明确返回失败更新订单状态；触发「代付失败自动驳回」逻辑
代付异常渠道超时/返回异常，结果不确定更新订单状态；建议人工核实

10.2 订单状态（已审核出款订单）

状态含义
已完成代付成功 / 人工完成，订单结束
已拒绝审核员拒绝，金额已退回会员余额
已罚没审核员罚没，金额不退回，不计入报表

10.3 处理方式

值说明
系统审核命中免审条件，系统自动通过并代付，操作人显示为 system
人工审核由审核员人工处理
PIX代付标注该订单通过 PIX 方式完成代付

10.4 渠道下架标准

值说明
不自动下架（推荐）渠道永久保留在自动代付可用池中，需人工手动下架
余额低于阈值时自动下架渠道余额低于配置的数值时，自动从可用池移除
连续失败超过次数时自动下架渠道连续代付失败次数超过配置值时，自动从可用池移除

11. 非功能性需求

  • 权限控制：锁定机制依赖账号权限体系，需明确"最高权限账号"的定义与识别方式；
  • 审计日志：锁定/解锁/拒绝/罚没/人工完成/代付等操作均需记录操作人、操作时间、操作前后状态，便于事后追溯；
  • 并发安全：批量操作、代付余额校验需在服务端做二次校验，避免并发场景下超发/超额代付；
  • 时区展示：所有时间字段需明确展示时区（如 UTC-3），避免跨时区团队协作时产生歧义；
  • 大数据量分页：总出款记录等页面数据量可达数万条，需要后端分页 + 索引优化，避免全表扫描。
